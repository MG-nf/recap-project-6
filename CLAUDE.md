# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is a freshly-generated Django project (`django-admin startproject`) with no apps, no README, and no dependency manifest (requirements.txt / pyproject.toml) yet. Django 6.0.7 is installed in the active Python environment but not pinned anywhere in the repo — if you add dependencies, create a requirements file and record the version.

## Architecture

- `config/` is the Django project package (created via `startproject`), not an app — settings, root URL conf, and WSGI/ASGI entrypoints live here (`config/settings.py`, `config/urls.py`, `config/wsgi.py`, `config/asgi.py`).
- No Django apps exist yet. New features should be added as apps via `python manage.py startapp <name>`, then registered in `INSTALLED_APPS` in `config/settings.py` and wired into `config/urls.py` with `include()`.
- Database is SQLite (`db.sqlite3` at the repo root), configured in `config/settings.py`.
- `SECRET_KEY` in `config/settings.py` is the insecure default generated by `startproject` — do not ship this to production as-is.

## Common commands

Run all commands from the repo root (where `manage.py` lives).

```bash
python manage.py runserver          # start the dev server
python manage.py migrate            # apply migrations
python manage.py makemigrations     # create migrations after model changes
python manage.py createsuperuser    # create an admin user
python manage.py test               # run the test suite
python manage.py test <app_label>   # run tests for a single app
python manage.py shell              # interactive shell with project loaded
```

There is no configured linter, formatter, or CI yet.

## Ticket-processing pipeline

Feature work in this repo goes through a Claude Code skill/agent pipeline defined under `.claude/`, driven by a state machine persisted in `WORKFLOW_STATE.md` (fields: `TicketID`, `State`, `Branch`, `LastUpdated`).

- `/process-ticket` is the entry point and orchestrates the whole flow: `/init-ticket` → `/refine` → `/plan` → `/implement` → `/review`, then (after manual approval) commits, pushes, merges the feature branch into `main`, and moves the ticket across the GitHub Project board columns (Ready → In progress → In review → Done).
- State transitions: `idle`/`done` → `initialized` → `refined` → `planned` → `implemented` → `reviewed` → `done`. Each skill in `.claude/skills/*/SKILL.md` only proceeds if `WORKFLOW_STATE.md` is in the state it expects — check/update that file before doing manual work in the middle of a ticket.
- `/init-ticket` creates the ticket's requirements doc (`plan/ticket-[id].md`) and a `feat/ticket-[id]-[sanitized-title]` branch, recorded as `Branch` in `WORKFLOW_STATE.md`. `/implement` checks the current git branch against that field and stops rather than switching branches itself.
- `/refine` and `/review` delegate to four read-only subagents in `.claude/agents/`: `security-reviewer`, `route-reviewer`, `service-reviewer`, `data-reviewer`. They only read/grep/glob (no `Edit`/`Write`/`Bash`) and report as a `[File Path] | [Issue Type] | [Brief Description]` table per `.claude/rules.md` rule 4.
- Pipeline artifacts accumulate under `plan/`: `ticket-[id].md`, `refinement-[id].md`, `feature-[id]-plan.md`, `review-[id].md`.
- `.claude/rules.md` are global rules referenced from here and apply regardless of which skill is active — notably: use the `gh` CLI for all GitHub issue/project-board interaction (no scraping); once API work starts, use Django REST Framework rather than Django Templates, with Route/Service/Data layers kept separate (the subagents above audit for that separation); and report findings as a 3-column table.
- Workflow Constraints (also in `.claude/rules.md`) codify this as a single-developer, no-PR project: merges go directly into `main` via local `git merge` + `git push origin main` (never `gh pr create`), and feature branches are intentionally never deleted after merging — kept as a historical record.
- `.claude/settings.json` scopes what the pipeline can actually run: `git status/diff/add/commit`, `gh *`, `python manage.py *`, and `WebFetch` are pre-allowed; `git push` and `docker` require a confirmation prompt each time; `curl`, `wget`, `rm`, and reads of `~/.ssh`, `~/.aws`, `.env` are denied outright.
