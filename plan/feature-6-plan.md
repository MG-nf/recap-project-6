# Feature Plan: Ticket 6 — Dashboard, UI, & Reporting

Based on `plan/ticket-6.md` and `plan/refinement-6.md`. One step per acceptance criterion. This is by far the largest ticket to date — the project's first server-rendered HTML, covering a Dashboard plus full CRUD templates for three existing models plus navigation plus (new) browser auth.

## Decisions (confirmed with user)

- **Auth gap**: add template-based **login and logout** (no signup template — JSON signup stays the only path for creating a new account, matching the ticket's ACs which don't mention Accounts as a UI-covered feature). Reuses `accounts/services.py::authenticate_user` + `django.contrib.auth.login()`/`logout()`, same as the existing JSON `LoginView`/`LogoutView`.
- **Duration-by-tag scope**: **per-goal breakdown** — each of a user's goals shows its own tag → duration totals on the Dashboard, not one number merged across all goals.
- **Duration-by-week scope**: dashboard-wide (across all of a user's goals) — this wasn't itself flagged as ambiguous (only the tag report was, due to the `JSONField` issue); "total duration per week" naturally reads as one user-wide activity timeline.
- **Untagged sessions**: shown as an explicit **"Untagged"** bucket in each goal's tag breakdown, so total duration always reconciles.
- **Styling**: plain authored CSS, no framework/CDN dependency.
- **App placement**: new `dashboard` app (added to `INSTALLED_APPS`) owns every template, static asset, and template-view route — `accounts`/`goals` stay pure-DRF/JSON, untouched except for the new reporting service functions living in `goals/services.py` (where the data already lives).
- **URL namespace**: root-level, via `path('', include('dashboard.urls', namespace='dashboard'))` in `config/urls.py` — entirely separate from the existing `/api/` prefix, no collision.
- **View pattern**: plain function-based views in `dashboard/views.py`, decorated with `@login_required` (Django's built-in; `LOGIN_URL = 'dashboard:login'` added to `config/settings.py` so an unauthenticated visit redirects to the login page instead of a raw 403/404). Each view calls the existing `goals/services.py`/`accounts/services.py` functions directly and translates their exceptions to `Http404`/redirect-with-message — no Django generic CBVs (`ListView`/`CreateView`/etc.), since those assume direct queryset/model-form access incompatible with this project's service-function pattern.
- **Forms**: plain `forms.Form` subclasses in `dashboard/forms.py` (not `ModelForm`, for the same service-layer reason) — field sets mirror the existing serializers (`GoalForm`: title/desc/status; `LearningSessionForm`: goal/date/duration/notes/tags; `ResourceForm`: goal/title/url/type). `LearningSessionForm.tags` is a comma-separated text input, split into a list before calling `create_session`/`update_session` — a form-layer concern only, no service-layer change.
- **Templates/static layout**: top-level `templates/` and `static/` directories, registered via `TEMPLATES[0]['DIRS']`/`STATICFILES_DIRS` in `config/settings.py`, so `base.html` can be extended by every app's templates without inter-app load-order coupling.
- **CSRF strategy**: Django forms + `{% csrf_token %}` throughout — template pages never call the JSON `/api/...` endpoints via `fetch`, keeping one consistent mechanism.
- **Scope per model**: `Goal` gets list + detail (a hub page showing its sessions/resources, mirroring the API's `resources_by_type`) + create + edit + delete. `LearningSession`/`Resource` get list (filterable by `?goal=<id>`, mirroring the API) + create + edit + delete — no separate detail page, since editing happens directly from the list.

## URL names (namespace `dashboard`)

`index` (`/`), `login` (`/login/`), `logout` (`/logout/`), `goal-list` (`/goals/`), `goal-detail` (`/goals/<pk>/`), `goal-create` (`/goals/create/`), `goal-edit` (`/goals/<pk>/edit/`), `goal-delete` (`/goals/<pk>/delete/`), and the same `list`/`create`/`edit`/`delete` shape for `session-*` (`/sessions/...`) and `resource-*` (`/resources/...`).

## Step 1 — AC1: "Dashboard provides a visual overview of goal counts grouped by status"

- **Test:** `dashboard/tests.py::DashboardGoalStatusTest` — authenticated `GET /` includes each status's count matching the user's actual goals (`planned`/`in-progress`/`done`); a status with zero goals shows 0, not an absent key; another user's goals never affect the counts; unauthenticated `GET /` redirects to `/login/`.
- **Implementation (also includes one-time infra for the whole ticket):** `python manage.py startapp dashboard`; add `dashboard` to `INSTALLED_APPS`; `config/settings.py`: `TEMPLATES[0]['DIRS'] = [BASE_DIR / 'templates']`, `STATICFILES_DIRS = [BASE_DIR / 'static']`, `LOGIN_URL = 'dashboard:login'`; `config/urls.py`: `path('', include('dashboard.urls'))`; `templates/base.html` (nav bar shell, `{% block content %}`); `dashboard/urls.py`/`dashboard/views.py` with `app_name = 'dashboard'`; `goals/services.py::goal_counts_by_status(user)` — `goals_for_user(user).values('status').annotate(count=Count('id'))`, mapped into a dict keyed by every `Goal.Status` value (so zero-count statuses appear); `dashboard/views.py::index(request)` (`@login_required`) assembles this into template context; `templates/dashboard/index.html` renders it as a simple table/bar list, extending `base.html`. Also included here since nothing else works without it: `dashboard/views.py::login_view`/`logout_view`, `templates/dashboard/login.html`, wired to `dashboard:login`/`dashboard:logout`.

## Step 2 — AC2: "Dashboard displays calculated reporting, including total duration hours per tag and per week"

- **Test:** `dashboard/tests.py::DashboardReportingTest` — a goal with tagged and untagged sessions shows correct per-tag duration totals plus an "Untagged" bucket for the untagged ones; week totals correctly sum durations across all of a user's goals/sessions grouped by week; another user's sessions never appear in either report.
- **Implementation:** `goals/services.py::duration_by_tag_for_goal(goal)` — fetches `goal.sessions.all().values('duration', 'tags')`, tallies duration per tag in Python, buckets sessions with an empty `tags` list under `"Untagged"`. `goals/services.py::duration_by_week_for_user(user)` — `sessions_for_user(user).annotate(week=TruncWeek('date')).values('week').annotate(total=Sum('duration')).order_by('week')`. `dashboard/views.py::index` extended to assemble both (the tag report iterates the user's goals, calling `duration_by_tag_for_goal` per goal); `templates/dashboard/index.html` extended with a per-goal tag-breakdown section and a per-week table.

## Step 3 — AC3: "Full UI Coverage: Every feature implemented in the project (Goals, Sessions, Resources) has a functional, intuitive template associated with it"

- **Test:** `dashboard/tests.py::GoalTemplateViewsTest`/`SessionTemplateViewsTest`/`ResourceTemplateViewsTest` — for each model: authenticated list/detail (Goal only) pages render 200 with the user's own data; create form submission (POST) creates the record and redirects to its list/detail page; edit form submission updates it; delete confirmation removes it; another user's records are never visible/editable/deletable (404, mirroring `get_goal_for_user`'s existing isolation); a `LearningSessionForm` submission with `tags` as `"reading, video"` correctly splits into `["reading", "video"]`; unauthenticated access to any of these redirects to login.
- **Implementation:** `dashboard/forms.py`: `GoalForm`, `LearningSessionForm` (with tag-splitting `clean_tags`), `ResourceForm`. `dashboard/views.py`: function-based `goal_list`/`goal_detail`/`goal_create`/`goal_edit`/`goal_delete` (and the session/resource equivalents, list/create/edit/delete only), each `@login_required`, each calling the existing `goals/services.py` CRUD functions (`list_goals_for_user`, `create_goal`, `get_goal_for_user`, `update_goal`, `delete_goal`, and session/resource equivalents) and catching their existing exceptions (`Goal.DoesNotExist`/`GoalNotOwnedError`/etc.) into `Http404`/a redirect with a Django messages-framework error. `dashboard/urls.py` wired per the URL-name list above. `templates/goals/{list,detail,form,confirm_delete}.html`, `templates/sessions/{list,form,confirm_delete}.html`, `templates/resources/{list,form,confirm_delete}.html`, all extending `base.html`.

## Step 4 — AC4: "Navigation is maintained, allowing intuitive movement between the Dashboard, Goals, Sessions, and Resources"

- **Test:** `dashboard/tests.py::NavigationTest` — every rendered page (Dashboard, Goal list/detail, Session list, Resource list) includes nav links to all four sections, generated via `{% url %}` (so a broken/renamed URL name fails the test rather than silently rendering a dead link); nav shows "Login" when logged out and "Logout" (plus the other three links gated behind auth) when logged in.
- **Implementation:** `templates/base.html`'s nav bar block, included on every page via `{% extends "base.html" %}`; `{% url 'dashboard:index' %}`/`'dashboard:goal-list'`/`'dashboard:session-list'`/`'dashboard:resource-list'`/`'dashboard:login'`/`'dashboard:logout'` — no hardcoded paths.

## Step 5 — AC5: "The overall UI is functional, accessible, and follows the unified navigation structure"

- **Test:** `dashboard/tests.py::AccessibilityTest` — every form field has a `<label for="...">` matching its input's `id`; every page has exactly one `<h1>`; nav links are real `<a href="...">` elements (keyboard-reachable, not JS-only click handlers). Full end-to-end verification (visually navigating Dashboard → Goal → Session/Resource → back, confirming forms submit correctly, confirming logged-out redirects) done manually via Playwright against the running dev server at `/review`, same established practice as tickets 2-5.
- **Implementation:** `static/dashboard/css/style.css` — semantic HTML structure, labeled form fields, visible focus states, adequate color contrast; no JS framework, no CDN dependency, consistent with this project's minimal-dependency history.
